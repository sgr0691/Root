# Root Technical Specification

## Overview

Root is a Rust CLI that wraps Nix to provide a simplified, deterministic, rollback-friendly package management experience for developers and AI coding agents.

Root should avoid reimplementing Nix internals in v0.1. Instead, it should invoke the Nix CLI through a controlled abstraction layer, manage Root-specific config/lock/snapshot metadata, and expose human-readable and JSON command outputs.

## Language

Root should be built in Rust.

Reasons:

- Strong CLI ecosystem
- Reliable filesystem/process control
- Good binary distribution story
- Strong typing for lockfiles and command outputs
- Fits infra/devtool audience

## Initial Platform

Root v0.1 targets:

- macOS
- Apple Silicon
- CLI packages
- Nix installed locally

Intel macOS, Linux, and Windows are post-MVP.

## Suggested Rust Crates

- `clap` for CLI parsing
- `serde` and `serde_json` for JSON output
- `toml` for Rootfile parsing
- `anyhow` or `thiserror` for errors
- `tokio` for async process execution if needed
- `tracing` for logs
- `console` or `indicatif` for terminal UX
- `which` for binary discovery
- `duct` or standard `std::process::Command` for invoking Nix

## Root Directories

```txt
~/.root/
  Rootfile
```

Project-level support is post-MVP, but Root should not block future project Rootfiles.

Future project-level paths:

```txt

```

## Rootfile Format

Rootfile is user-authored or generated by Root.

Example:

```toml
[packages]
node = "22.11.0"
postgresql = "16.4"
poppler = "24.08.0"
ffmpeg = "7.1"
ripgrep = "14.1.1"

[settings]
snapshots = true
verify_installs = true
```

## root.lock Format

The lockfile is generated and should include exact resolved package metadata.

Example shape:

```json
{
  "version": 1,
  "platform": "aarch64-darwin",
  "nixpkgs": {
    "rev": "<resolved-revision>",
    "source": "github:NixOS/nixpkgs"
  },
  "packages": [
    {
      "name": "poppler",
      "requested": "poppler",
      "version": "24.08.0",
      "attribute": "poppler",
      "storePath": "/nix/store/...-poppler-24.08.0",
      "binaries": ["pdftotext", "pdfinfo"]
    }
  ]
}
```

## Command Output Contract

Every command should support:

```bash
--json
```

Human output should be beautiful and concise.

JSON output should be stable, typed, and friendly to coding agents.

Example:

```json
{
  "success": true,
  "operation": "install",
  "package": "poppler",
  "changed": ["poppler"],
  "unchanged": ["node", "ruby", "postgres", "python"],
  "snapshot_id": "snap_20260524_153000",
  "rollback_available": true,
  "warnings": []
}
```

## Nix Integration Strategy

MVP should invoke Nix directly.

Possible commands:

```bash
nix search nixpkgs poppler
nix profile install nixpkgs#poppler
nix profile list
nix profile remove <index|package>
```

Root should hide Nix errors and convert them into Root-native messages.

Bad:

```txt
attribute 'aarch64-darwin' missing from derivation
```

Good:

```txt
This package is not available for your Mac architecture.
Try `root search poppler` to find alternatives.
```

## Snapshot Model

A snapshot records Root-managed state before a mutation.

Snapshot metadata:

```json
{
  "id": "snap_20260524_153000",
  "created_at": "2026-05-24T15:30:00-04:00",
  "reason": "before install poppler",
  "rootfile_hash": "...",
  "lockfile_hash": "...",
  "packages": [...]
}
```

Rollback should restore the prior Root-managed package state. MVP rollback may be implemented by restoring prior lockfile metadata and reconciling the Nix profile.

## Doctor Checks

`root doctor` should detect:

- Nix installed or missing
- Root profile available
- Root-managed binaries on PATH
- unmanaged binaries shadowing Root binaries
- lockfile drift
- missing packages
- version mismatch
- Brew conflicts for common tools

## Brew Import

`root import brew` should:

1. Run `brew list --formula` if Brew exists.
2. Run `brew list --cask` but mark casks as unsupported for MVP.
3. Attempt to map formula names to nixpkgs attributes.
4. Generate a report.
5. Optionally generate a Rootfile candidate.

MVP should be conservative and avoid automatic migration unless user explicitly confirms.

## Verify Command

`root verify <pkg>` should check:

- package exists in root.lock
- expected binary exists on PATH or Root profile path
- binary executes with `--version`, `-version`, or a known verification command
- optional package-specific verification metadata

## Security Principles

- Never run `curl | sh` from agent workflows.
- Never mutate shell profiles without explicit user confirmation.
- Never hide system changes.
- Always create a snapshot before mutation.
- Keep Root-managed PATH isolated and inspectable.
- Prefer fail-closed behavior for agent-driven installs.

## Agent Compatibility

Root should be agent-native via:

- JSON output
- deterministic exit codes
- explicit plan/install/verify/rollback flow
- skill files for common agents
- clear no-Brew/no-global-mutation protocol

## Exit Codes

Suggested:

```txt
0 success
1 generic failure
2 invalid arguments
3 package not found
4 verification failed
5 drift detected
6 rollback failed
7 nix unavailable
8 unsupported platform
```
